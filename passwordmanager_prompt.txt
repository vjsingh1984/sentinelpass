# PROMPT: SECURITY + PRODUCT GAP AUDIT AND EXECUTION PLAN

You are a principal security engineer, product architect, and staff engineer.

You are reviewing an existing local-first password manager codebase (Rust core + desktop app + browser extension + optional sync relay).

Your job is NOT to produce a greenfield fantasy design.
Your job is to audit the ACTUAL implementation, identify gaps, and produce an actionable plan.

## Non-negotiable rules

1. Distinguish `implemented`, `partial`, and `planned` for every major control/feature.
2. Cite concrete evidence from the codebase (file paths and line references when available).
3. Do not claim a mitigation is present unless you can point to code or tests.
4. Separate:
   - security posture gaps,
   - product completeness gaps,
   - design/architecture gaps,
   - implementation/testing gaps,
   - documentation/governance gaps.
5. Include open-source vs free vs paid tier recommendations.
6. Include a strategy for maintaining a public OSS repo and a private commercial repo without a long-lived fork.

## Deliverables (in this order)

### 1. Executive Gap Review

Produce a prioritized gap review with:

- severity (`critical`, `high`, `medium`, `low`)
- category (`security`, `feature`, `design`, `implementation`, `docs`, `commercialization`)
- evidence (file refs)
- user impact
- exploitation or failure mode
- recommended remediation

### 2. Security Posture Matrix

Create a matrix for key controls:

- local vault encryption
- KDF
- lockout
- audit logging
- clipboard handling
- extension origin validation
- daemon IPC transport security
- sync relay authentication
- replay protection
- relay rate limiting
- relay cleanup / data retention
- pairing endpoint abuse controls
- mobile secure storage / biometrics

For each: `Implemented / Partial / Planned` + evidence.

### 3. PRD (12-month)

Write a concise PRD covering:

- vision
- target users / personas
- problem statement
- differentiation
- product principles
- scope and non-goals
- free tier and paid tier packaging
- success metrics
- risks and dependencies

### 4. Requirements (traceable)

Create requirements with IDs and acceptance criteria, including:

- functional requirements
- security requirements
- operational requirements
- testing requirements
- commercialization / tiering requirements
- OSS/private repo governance requirements

### 5. Solution Design

Produce a design that addresses the identified gaps, including:

- trust boundaries
- extension -> native host -> daemon request validation
- Windows IPC hardening strategy
- relay hardening (authz, rate limits, cleanup, config wiring, TLS assumptions)
- feature flag / entitlement model
- plugin seams for paid features (without modifying core cryptographic trust)
- data/privacy and telemetry boundaries

### 6. Roadmap and Implementation Plan

Produce:

- 90-day remediation plan
- 6-12 month roadmap
- owners/workstreams (if names unknown, use roles)
- dependencies and sequencing
- release criteria
- test/security verification plan

### 7. Open Source / Free / Paid Strategy

Decide and justify:

- what should be open source
- what can be free but distributed as official binaries
- what should be paid/private
- what should NEVER be closed (trust-critical surfaces)
- two-repo operating model (public + private)
- versioning, release, CI, and compatibility strategy

## Additional constraints for the analysis

- Assume this is a security-sensitive application used for banking, healthcare, developer, and enterprise credentials.
- Prefer practical, incremental changes over rewrites.
- Call out doc/implementation mismatches explicitly.
- If a roadmap item is already implemented, mark the roadmap as stale and propose an updated replacement.

## Output format requirements

- Start with findings first (highest severity first).
- Use file references for findings.
- Keep summaries brief; details should be in the plan/design sections.
- Make the final output suitable to become repository docs with minimal editing.
